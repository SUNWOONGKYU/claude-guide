# Claude와 Claude Code 시리즈 작성 지침

## 문서 구조 패턴 (1-6편 분석 기준)

### 제목 형식
- 1편: `# Claude와 Claude Code의 이해 및 활용법` (전체 제목) + `## X편 | 제목`
- 2-6편: `# X편 | 제목` (바로 편 제목으로 시작)
- 구분선: 제목 다음 `---` 사용

### 섹션 구성
- 도입부: 주제 정의와 배경 설명 (별도 제목 없이 바로 시작)
- 본문: `## 숫자. 대제목` → `### 소제목` → `#### 세부제목` 구조
- 마무리: 작성 정보만 표시

### 내용 전개 방식
- 개념 설명 → 구체적 예시 → 실제 활용 순서
- 기술 용어는 첫 등장 시 괄호로 간단히 설명

## 작성 프로세스

### 작업 시작 전 확인 사항
- **사용자가 "작업을 하기 전에 질문하라"고 지시한 경우**: 필요한 질문을 모두 한 후, 답변을 받으면 작업 대기
- **명확한 작업 지시 필요**: 질문에 대한 답변을 받았어도, 사용자가 명확하게 "작업 시작", "작성해", "진행해" 등의 지시를 해야만 작업 수행
- **작업 보류**: 답변만 받고 작업 지시가 없으면 대기 상태 유지

## 작성 원칙

### 1. 균형잡힌 접근
- 지시사항을 기본 틀로 삼되, 창의적 표현 활용
- 기존 시리즈의 스타일과 톤을 유지하면서 새로운 가치 전달
- 독자에게 실질적 도움이 되는 내용 구성

### 2. 일관성 유지
- 1-6편의 구조와 스타일을 참고하여 통일성 확보
- 독자가 익숙하게 느낄 수 있는 형식 사용
- 시리즈 전체의 흐름을 고려한 내용 전개

### 3. 명확한 구조
- 도입부 → 본문 → 작성 정보의 간결한 구성
- 본론 중심의 실용적 내용 전개
- 불필요한 중복이나 미사여구 배제

### 4. 실용적 내용
- 구체적 예시와 실제 명령어 제공
- 개발자/비개발자 관점 구분 (해당 시)
- 현실적 한계와 제약사항 명시
- 코드 예시는 이해를 위해 꼭 필요한 경우에만 10줄 이내로 제시

### 5. 적절한 톤
- 친근하면서도 전문적인 어조
- "~입니다", "~합니다" 정중체 사용
- 비유와 예시를 통한 쉬운 설명
- 굵은 글씨(**) 적절히 사용하여 핵심 강조
- 핵심, 완전 이런식의 단어 남발 금지

## 개발자/비개발자 구분 상세 지침

### 언제 구분이 필요한가
- **필수적인 경우**: 기술적 깊이가 크게 다른 내용을 다룰 때
- **선택적인 경우**: 같은 기능을 다른 관점에서 설명할 때
- **불필요한 경우**: 모든 독자에게 동일하게 적용되는 내용

### 구분 방법
1. **명시적 구분**: 소제목으로 "개발자를 위한", "비개발자를 위한" 명확히 표시
2. **내용 차별화**:
   - 개발자: 기술 용어, 간단한 코드 예시(10줄 이내), 고급 설정 중심
   - 비개발자: 일상 용어, 실생활 비유, 기본 활용 중심
3. **분량 균형**: 두 그룹 모두에게 충분한 가치 제공

### 주의사항
- 모든 주제에 강제로 구분을 적용하지 말 것
- 자연스럽게 통합 가능한 내용은 굳이 나누지 않기
- 한 그룹을 소외시키지 않도록 균형 유지
- "개발자는 이미 알 것이다" 같은 가정 피하기

### 실제 활용 예시
**좋은 예**: 5편의 파일 시스템 접근
- 개발자 섹션: git 명령어, 코드 리팩토링, 테스트 자동화
- 비개발자 섹션: 파일 정리, 문서 작성, 데이터 분석

**나쁜 예**: 강제적 구분
- "개발자는 A 기능을, 비개발자는 B 기능을 사용하세요" (X)
- 모든 독자가 활용할 수 있는 공통 기능을 굳이 나누기 (X)

## HTML 파일 작성 시 준수사항

### 원칙
- **원본 마크다운 파일의 내용을 충실히 반영**
- 마크다운에 없는 내용을 임의로 추가하지 않음
- 특히 도입부(intro-highlight)는 원본에 있을 때만 포함

### 구체적 지침
1. **콘텐츠 충실성**
   - 마크다운 파일의 내용을 그대로 HTML로 변환
   - 새로운 문단이나 설명을 창작하지 않음
   - 원본에 없는 도입부를 만들지 않음

2. **스타일 적용**
   - 디자인과 스타일은 기존 HTML 파일 참고
   - 레이아웃과 구조는 통일성 유지
   - 색상, 폰트, 박스 스타일 등은 시리즈 일관성 유지

3. **검증 방법**
   - HTML 작성 전 원본 마크다운 파일 구조 확인
   - 특히 도입부 존재 여부 체크
   - 모든 내용이 원본에 근거하는지 확인
